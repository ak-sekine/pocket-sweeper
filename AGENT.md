# Pocket Sweeper AI作業ガイド

## プロジェクトの正本

- Pocket Sweeperは、初代ゲームボーイ向けマインスイーパーです。
- `PROJECT.md` はプロジェクト概要とWBSの正本です。WBSのチェック状態は、実施した作業だけを更新します。
- `docs/` 配下は分野別仕様書です。仕様の詳細は対象分野の仕様書を正本とします。
- `README.md` は環境構築・ビルド・基本的な使い方の入口です。

## 作業開始時

1. `git status --short --branch` と最新コミットを確認する。
2. リモートの最新状態を取得し、未コミット変更を確認したうえで必要なら `git pull --ff-only` を実行する。
3. `PROJECT.md`、対象WBS、およびWBSから参照される仕様書を読む。
4. 関連する実装、テスト、既存資産、確認手順書を調査する。
5. 作業開始時点のブランチ名、HEAD、作業ツリーの変更を記録する。

仕様を推測で補完しない。仕様が見つからない場合は未確定事項として報告し、仕様書同士に矛盾がある場合は勝手に解消しない。

## WBS運用

- WBSの親項目は、子孫項目がすべて完了した場合だけ完了にする。
- AIだけで完了できる作業と、人による確認が必要な作業を区別する。
- 人による確認が必要な場合は、WBSを「AIによる準備・自動確認」と「人による確認」に分割する。
- AI側だけ完了した場合は、AI側だけを完了にし、人側と親項目は未完了のままにする。
- 人による確認結果が提供され、確認項目が成功している場合は、人側と親項目を完了にする。
- 今回の作業と無関係なWBS、後続WBS、仕様書のチェック状態は変更しない。

## 仕様・実装・変更範囲

- 詳細仕様は仕様書へ記載し、同じ仕様をPROJECT.mdや別文書へ重複記載しない。
- 仕様変更が必要な場合は正本の仕様書を更新する。単なる確認結果は新しい仕様として記載しない。
- 既存のJSON、生成物、テスト、未コミット変更を無断で書き換えない。
- 変更は対象WBSに直接関係するファイルだけに限定する。
- 無関係なリファクタリング、整形、命名変更を行わない。
- 既存の変更と今回の変更が混在して安全に分離できない場合は、無理にコミットしない。

## テストと確認

- 変更内容に応じた既存テストを確認し、標準テストコマンドを実行する。
- 生成物は生成成功だけでなく、仕様に対応する主要構造や既存の期待値と照合する。
- 自動確認と人によるGUI・実機・エミュレータ確認を区別する。自動確認だけで人手確認を完了扱いにしない。
- テストが失敗した場合は、原因を確認して修正するまでコミット・pushしない。
- 文書のみの変更でも `git diff --check` を実行する。

## Git運用

WBSに伴う実装・確認作業では、原則としてテスト、commit、pushまで実施する。調査のみ、プロンプト作成のみ、またはユーザーがGit操作を禁止した場合はcommit・pushしない。

- 対象ファイルを明示的にstageする。`git add .` と `git add -A` は使わない。
- コミットメッセージは変更内容が分かるものにする。
- force push、reset、rebase、clean、checkoutによる変更破棄などの破壊的操作は禁止する。
- pushがnon-fast-forwardで拒否された場合は、force pushせず状況を報告する。
- 作業後に `git status --short`、`git diff --check`、`git diff --stat`、必要に応じて`git diff`を確認する。

## 最終報告

最低限、次を報告する。

- 実施内容と確認範囲
- 変更ファイル
- テストコマンドと結果
- 未確認事項・残課題
- ブランチ、コミットハッシュ、コミットメッセージ
- push先とpush結果

## ツール固有の補足

- Pythonツールは原則としてプロジェクトの`.venv`を使用し、利用できない場合は環境の`python3`を使用する。
- RGBDS、SameBoy、BGB、hUGETrackerなど外部ツールを実行できない場合は、実行したと報告せず、未確認範囲と理由を記録する。
- hUGETrackerやエミュレータのGUI確認は、人が実際に操作した結果が提供された場合だけ完了扱いにする。
